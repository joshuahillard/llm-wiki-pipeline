Strategic assessment: Managed Agents vs. Agent SDK for the validation pipeline — April 8, 2026

1. What Just Shipped

Anthropic launched Claude Managed Agents (beta, managed-agents-2026-04-01) alongside the existing Claude Agent SDK. These are two distinct products for building autonomous agents on Claude.

Managed Agents

A hosted service. You define an agent (model + system prompt + tools) and an environment (cloud container with packages and networking). Anthropic runs it. Sessions persist through disconnections. You stream events via SSE.

Pricing: Standard token cost + $0.08/session-hour active runtime + $10/1K web searches.

Think of it as: "Claude Code as a cloud service you configure via API."

Agent SDK

A library (Python / TypeScript) that gives you the same agent loop and tools as Claude Code, but running on your infrastructure. You control the process, the sandbox, the filesystem.

Pricing: Standard API token cost only. You own the compute.

Think of it as: "Claude Code as a Python import."

4. Three Possible Plays

Play A: Agent SDK as the Next Orchestrator

Recommended evolution path

Replace Run-Validator.ps1 with a Python Agent SDK wrapper. The agent reads articles from provisional/, orchestrates evaluation, writes structured results, and updates the ledger.

  • TD-001 path — Two options: (1) Agent-loop evaluation where Claude reasons directly (requires golden corpus benchmark to validate fidelity), or (2) Agent orchestrates but delegates evaluation to a clean Messages API call, preserving prompt isolation. Option 2 is lower-risk.
  • Preserves local state and hybrid deployment path (Bedrock/Vertex official; self-hosted via custom abstraction)
  • Hooks provide audit logging and governance enforcement points; implementation of exact EXIT_TOKEN_OVERFLOW semantics needs a custom layer on SDK primitives
  • Test suite runs unchanged
  • Subagents could split: one for evaluation, one for promotion prep

Play B: Managed Agents for Céal or Future Projects

Better fit for other work

Managed Agents excels for long-running, cloud-native tasks where you don't need fine-grained state control. Céal (job automation) or a future content generation pipeline would be natural fits — tasks where the agent runs for minutes, does web research, generates artifacts, and reports back.

  • No infrastructure to manage
  • Web search + fetch built in
  • Custom tools for external APIs (job boards, etc.)
  • Sessions persist through disconnections

Play C: Hybrid — SDK Now, Managed Later

Evolutionary path

Start with the Agent SDK for LLM-Wiki (Play A). The Agent SDK is the safer default unless Managed Agents session persistence, event export, and externalized ledger writes are proven sufficient for the governance model. As Managed Agents matures out of beta — especially cross-session storage durability and multi-agent support — re-evaluate. The system prompt, policy bundle, and tool definitions would transfer directly; only the state management and evaluation isolation patterns need validation.

5. Baseline Alternative: Keep Current Orchestrator, Add Messages API Provider

Play 0: Wire the Existing Pipeline to the Messages API

Lowest-risk path

Before evaluating either agent model, the most conservative TD-001 resolution is: keep Run-Validator.ps1 and validator_runner.py exactly as designed, and implement a live provider that calls the Anthropic Messages API (or Bedrock/Vertex equivalent) in place of the stub. No orchestrator change. No new runtime. No agent loop.

When Play 0 is the right call: If the near-term goal is shipping a working live provider with minimal risk, Play 0 is the correct move. It closes TD-001 cleanly, preserves every existing invariant, and creates the provider abstraction layer that both the Agent SDK path and the self-hosted path need anyway.
When Play A adds value over Play 0: If the pipeline needs to grow beyond deterministic script execution — e.g., the evaluator needs to reason about article context, fetch references, handle ambiguous policy edge cases, or coordinate multi-step promotion workflows — then the Agent SDK provides capabilities that a Messages API call cannot. Play 0 is the foundation; Play A is the evolution.

These are not mutually exclusive. The disciplined path is: Play 0 first (wire the provider, close TD-001, validate against the golden corpus), then Play A as a follow-on if autonomous reasoning adds measurable value to evaluation quality or pipeline flexibility.

6. What Would Change in LLM-Wiki (Play A)

If you went with Play A (Agent SDK):
  • Run-Validator.ps1 → run_validator_agent.py using claude_agent_sdk.query()
  • validator_runner.py → Recommended: keep as a structured tool the agent calls via a clean Messages API evaluation call (prompt isolation preserved). Alternative: collapse into agent reasoning, but benchmark against golden corpus first
  • Stub provider → TD-001 gets a concrete implementation path, not an automatic close. Fidelity benchmark is the decision gate.
  • Token budgeting → SDK provides max_budget_usd, max_turns, per-step usage, and hook inspection points. EXIT_TOKEN_OVERFLOW (code 5) semantics require a custom enforcement layer on these primitives.
  • Ledger writes → The agent must not write the ledger directly. Options: (1) PreToolUse hook that intercepts Edit/Write calls targeting the ledger path and rejects any non-append mutation, (2) a dedicated append_ledger custom tool that enforces append-only semantics server-side, or (3) an external append-only store (e.g., a local service or file-lock wrapper) that the agent cannot mutate directly. PostToolUse is insufficient — it runs after the write succeeds and cannot prevent or roll back a bad mutation.
  • Gitea PR → Custom MCP server or tool definition wrapping Gitea API. TD-002 gets a clean implementation path.
  • Human-in-the-loop → AskUserQuestion tool or permission gate before promotion
  • Test suite → Unchanged. 196 assertions still run locally via pytest.
  • Hybrid deployment → SDK supports Bedrock, Vertex, Azure natively. Self-hosted model fallback requires custom provider abstraction (not SDK-native).

7. Decision Gates and Open Questions

Decision Gates (must resolve before committing)

  • Evaluation fidelity Gating — Agent-loop evaluation changes the prompt stack, tool context, and execution semantics versus a direct Messages API call with the same prompt. Before treating agent-loop evaluation as a TD-001 replacement, benchmark both approaches against the 42-fixture golden corpus. If fidelity diverges, the clean-call path (agent orchestrates, Messages API evaluates) becomes mandatory, not optional.
  • Managed Agents session persistence semantics Gating for Play C — Managed Agents documents session-persistent filesystems. The unresolved question is whether this persistence survives session boundaries, supports append-only write semantics, and provides export/backup guarantees sufficient for the ledger invariant. If yes, Play C's timeline accelerates significantly.

Open Questions (inform but don't block)

  • Cost modeling: A full pipeline run involves ~N articles × K tokens per evaluation. The Agent SDK adds overhead from the agent loop (tool calls, reasoning tokens) on top of evaluation tokens. Model the per-article cost delta for both collapsed and separated evaluation paths.
  • Prompt architecture compatibility: The existing persona library and mode packs are designed for direct prompting. How do they map onto agent system prompts and Skills? The SDK's Skills support (.claude/skills/) is a natural fit for the policy bundle.
  • EXIT_TOKEN_OVERFLOW implementation: SDK provides max_budget_usd, max_turns, and per-step usage. An exact pre-generation hook reporting future token totals is not documented. The overflow semantics need a custom enforcement layer — design this before assuming budget control is solved.

This conversation documents the iterative architectural hardening of an automated **LLM-driven Content Validation & Promotion Pipeline**. The goal was to transform a set of functional PowerShell and Python scripts into an enterprise-grade, deterministic control plane for a wiki-style repository.

The evolution focused on three core pillars: **Identity, Idempotency, and Integrity.**

## ---

**1\. The "Rules-of-Reality" Digest**

To prevent "stale" validation states, we replaced static versioning with a **Runtime Composite Digest**. This ensures that if the logic of the system changes, the cache automatically invalidates.

* **The Mechanism:** A SHA256 hash is generated by concatenating the content of every artifact that materially affects a transaction.  
* **The Inputs:**  
  * **Logic:** Run-Validator.ps1, Promote-ToVerified.ps1, validator\_runner.py.  
  * **Identity:** parse\_identity.py (The Single Source of Truth parser).  
  * **Rules:** validation\_result.schema.json and the \_policy\_bundle.md.  
  * **State:** The current origin/main commit SHA.  
* **The Result:** Any change to the prompt, the Python runner, or the Gitea base state changes the $ContextDigest, forcing the system to re-evaluate all pending drafts.

## **2\. Deterministic Transaction Identity**

We eliminated collisions and "split-brain" parsing by moving away from simple filename-based tracking to a **Canonical Transaction Key**.

* **Single Source of Truth (SSoT):** We moved all frontmatter parsing into a dedicated Python utility (parse\_identity.py). This prevents the PowerShell wrapper and Python runner from disagreeing on a file's source\_id due to regex variance.  
* **The Key Structure:** \> source\_id \+ repo\_relative\_path \+ document\_hash \+ context\_digest  
* **Namespace Fidelity:** By using the repo-relative path, we ensure that files with the same name in different subdirectories (e.g., networking/dns.md vs. security/dns.md) never collide in the ledger or the verified output.

## **3\. Idempotency & The "Ghost Branch" Problem**

A major focus was ensuring that retrying a failed or interrupted promotion does not corrupt the workspace or the Git history.

* **From Commit SHA to Tree SHA:** Since Git commit SHAs change based on timestamps, we shifted to using **Tree SHAs** (HEAD^{tree}) to verify remote state. This allows the system to prove that a remote branch is physically identical to the local intent, even if the commit metadata differs.  
* **Fast-Path Cleanup:** We implemented a "look-before-you-leap" check. If an open PR already exists on Gitea, the script verifies the **Base SHA** and the **Tree SHA**. If they match, it updates the local ledger and performs a **Full Workspace Rollback** (resetting the branch, moving the file back to provisional, and deleting the local ephemeral branch) before exiting.  
* **Zero-Footprint Execution:** The system now cleans up orphaned local branches and empty directories even on successful "fast-path" exits, ensuring the automation clone remains pristine for the next run.

## **4\. Security & Promotion Workflow**

The promotion logic was hardened to respect protected-branch models while maintaining automation efficiency.

* **Force-Push Elimination:** We removed \--force from the workflow. The script now reconciles remote state explicitly. If a push is rejected, it fetches the remote and compares the Tree SHA to ensure no tampering has occurred before proceeding.  
* **Context-Bound Branching:** Ephemeral branches are now named using the $ContextDigest. This guarantees that a re-evaluation against new rules results in a new branch and a new PR, rather than overwriting an existing review state.

### ---

**Summary of Improvements**

1. **Eliminated State Leaks:** No orphaned branches, empty folders, or local-only commits left behind.  
2. **Hardened Identity:** Deterministic keys prevent collisions across renamed files or duplicate basenames.  
3. **Mathematical Integrity:** Tree SHA verification ensures the physical state of the PR matches the expected evaluation.  
4. **Automatic Invalidation:** The "Rules-of-Reality" digest makes the system self-correcting when policies or code change.

This architecture is now ready for deployment into a high-concurrency CI/CD environment, providing a defensible and auditable path for AI-generated content.
